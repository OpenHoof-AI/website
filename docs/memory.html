Memory System

OpenHoof gives your AI persistent memory using vector embeddings stored locally. Your agent remembers past conversations, user preferences, and important facts across sessions — without sending anything to the cloud.

How Memory Works

The memory system has three layers:

  1. Conversation buffer — The current chat session's messages, kept in RAM. This is the immediate context the AI sees.
  2. Short-term memory — Recent conversation summaries and extracted facts, stored in SQLite. Automatically pulled into context when relevant.
  3. Long-term memory — All stored facts and conversations as vector embeddings. Searched via semantic similarity when the AI needs to recall something.

When you send a message, OpenHoof queries long-term memory for semantically relevant entries and injects them into the agent's context. The AI sees these memories alongside your current message, giving it the ability to reference past discussions naturally.

Auto-Save Mode

When memory.autoSave is enabled (the default), OpenHoof automatically extracts and stores important facts from conversations. The extraction runs after each exchange and identifies:

  • User preferences and personal details
  • Decisions and agreements
  • Technical information and project context
  • Tasks and follow-ups
{
  "memory": {
    "enabled": true,
    "autoSave": true,
    "autoSaveThreshold": 0.7
  }
}

The autoSaveThreshold controls how aggressively facts are saved. Lower values (0.3–0.5) save more aggressively; higher values (0.7–0.9) only save clearly important information.

Manual Memory Management

You can also manage memory directly through the CLI:

# Add a memory entry manually
openhoof memory add "User prefers dark mode in all applications"

# Search memory
openhoof memory search "user preferences"

# List recent memory entries
openhoof memory list --limit 20

# Delete a specific memory entry by ID
openhoof memory delete mem_abc123

# Export all memories to a JSON file
openhoof memory export > memories.json

# Import memories from a JSON file
openhoof memory import memories.json

# Clear all memory (destructive!)
openhoof memory clear --confirm

Memory via Chat

Your AI agent can also manage its own memory during conversations. You can tell it to remember things:

You: Remember that my server IP is 192.168.1.100
AI: Got it, I've saved that to memory. I'll remember your server IP is 192.168.1.100.

You: What's my server IP?
AI: Your server IP is 192.168.1.100.

You: Forget my server IP
AI: Done, I've removed that from my memory.

The agent uses built-in memory tools to store, search, and delete entries during conversations.

Vector Embeddings

Memory entries are converted to vector embeddings for semantic search. OpenHoof uses a local embedding model by default, so nothing leaves your machine:

OptionDefaultDescription
memory.embeddingModel"all-MiniLM-L6-v2"The sentence-transformer model used for embeddings
memory.embeddingDimensions384Vector dimensions (must match the model)
memory.similarityThreshold0.65Minimum similarity score for memory retrieval
memory.maxResults10Maximum memories injected per query

The default model (all-MiniLM-L6-v2) is downloaded automatically on first use (~80MB). It runs on CPU and is fast enough for real-time memory lookups.

Using OpenAI Embeddings

For higher quality embeddings, you can use OpenAI's API instead:

{
  "memory": {
    "embeddingProvider": "openai",
    "embeddingModel": "text-embedding-3-small",
    "embeddingApiKey": "sk-..."
  }
}

Storage Backends

Local (Default)

Stores everything in SQLite with a vector index. Data lives at ~/.openhoof/memory.db. Zero configuration, works everywhere:

{
  "memory": {
    "backend": "local",
    "dbPath": "~/.openhoof/memory.db"
  }
}

PostgreSQL

For larger deployments or if you want to share memory across machines, use PostgreSQL with the pgvector extension:

{
  "memory": {
    "backend": "postgres",
    "postgresUrl": "postgresql://user:pass@localhost:5432/openhoof"
  }
}
ℹ️
PostgreSQL requires the pgvector extension. Install it with CREATE EXTENSION vector; in your database. OpenHoof creates the required tables automatically on first connection.

Memory Namespaces

You can organize memories into namespaces to separate concerns. This is useful if multiple people use the same OpenHoof instance or if you want per-project memory:

# Add to a specific namespace
openhoof memory add "Project uses React 19" --namespace "work/frontend"

# Search within a namespace
openhoof memory search "React version" --namespace "work/frontend"

In config, you can assign namespaces per channel:

{
  "channels": {
    "telegram": {
      "memoryNamespace": "personal"
    },
    "slack": {
      "memoryNamespace": "work"
    }
  }
}
